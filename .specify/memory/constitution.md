# holaMundo Constitution

## Core Principles

### I. Spec-Driven Delivery
Every feature MUST begin with a written spec in `spec.md` that defines prioritized user
stories, measurable success criteria, and explicit assumptions. Implementation plans and tasks
MUST derive from the approved spec; untracked work is non-compliant.
Rationale: Shared written intent reduces rework and keeps implementation aligned with product
outcomes.

### II. Independent Value Slices
Work MUST be organized as independently deliverable user stories (P1, P2, P3...). Each story
MUST define an independent test/validation path and MUST deliver user-visible value on its own.
Cross-story dependencies are allowed only when documented in `plan.md` with a clear reason.
Rationale: Independent slices enable incremental delivery and lower release risk.

### III. Verification-First Quality Gates
Before merge, every changed behavior MUST be verified by executable evidence: automated tests,
or when automation is not feasible, a documented manual validation procedure in tasks and
quickstart artifacts. Critical paths and regressions MUST be covered by automated tests.
Rationale: Reliable verification is the minimum bar for safe iteration.

### IV. Traceability by Default
Requirements, plan decisions, and implementation tasks MUST remain traceable from `spec.md` to
`plan.md` to `tasks.md`. Task entries MUST include concrete file paths and story labels.
Any requirement without an implementation task, or any task without requirement linkage,
MUST be resolved before execution.
Rationale: Traceability improves review quality and prevents scope drift.

### V. Simplicity and Change Discipline
Solutions MUST prefer the simplest architecture that satisfies current requirements. New
dependencies, abstractions, or structural complexity MUST include written justification in
`plan.md` (Complexity Tracking). Breaking governance changes MUST follow semantic versioning and
explicit amendment notes.
Rationale: Simplicity preserves maintainability and reduces hidden cost.

## Operational Constraints

- Artifacts generated by Speckit commands MUST keep placeholders out of final deliverables unless
	explicitly marked as deferred with `NEEDS CLARIFICATION` or `TODO(...)` and justified.
- Dates in governance and planning artifacts MUST use ISO format `YYYY-MM-DD`.
- Constitution checks in plans MUST be completed before Phase 0 research and re-checked after
	Phase 1 design.

## Delivery Workflow and Reviews

1. `spec.md` defines user stories, requirements, and measurable outcomes.
2. `plan.md` captures technical decisions, constitution gates, and structure choices.
3. `tasks.md` lists dependency-ordered tasks by user story with explicit paths.
4. Implementation proceeds by story priority, validating each story independently before moving on.
5. Reviews MUST verify constitution compliance and evidence of verification for changed behavior.

## Technology Stack - React Web

This project is a React-based web application with the following mandatory technology stack:

- **Framework**: React 18+ (or specified version in package.json)
- **Node.js**: Version 18 LTS or newer
- **Build Tool**: Vite or Next.js (must be specified in plan.md)
- **Package Manager**: npm 9+ or pnpm 8+
- **Language**: TypeScript (strict mode REQUIRED for all new code)
- **Code Quality**:
  - ESLint with Airbnb or Standard config
  - Prettier for code formatting (mandatory pre-commit hook)
  - Husky for git hooks enforcement
- **Styling**: Tailwind CSS or CSS-in-JS (document choice in plan.md)
- **State Management**: Context API or Redux/Zustand (justify if multi-store choice needed)
- **API Client**: Fetch API, Axios, or tRPC (specify in plan.md)
- **Routing**: React Router v6+ (for multi-page apps)
- **Environment**: Support for .env files with strict variable validation

No external packages may be added to core dependencies without Complexity Tracking
justification in plan.md.

## Component Architecture & Templates

React components MUST follow a standardized template-based architecture for consistency:

- **Folder Structure**: All components live in `src/components/` organized by feature, not type.
  Example: `src/components/UserProfile/`, `src/components/Dashboard/`
- **Template Requirements**:
  - Every component MUST have a `.tsx` file with named export
  - Every component MUST have a `.test.tsx` file (co-located)
  - Components using complex props MUST define TypeScript interfaces in `.types.ts`
  - Reusable hooks MUST live in `src/hooks/` with `use` prefix and `.ts` extension
- **Component Types**:
  - **Functional Components**: All components MUST use function declarations with FC or React.FC
  - **Presentation vs Container**: Document which pattern is used in component comments
  - **Compound Components**: MUST include examples in JSDoc comments
  - **Custom Hooks**: MUST document return type, dependencies, and side effects clearly
- **Template Standardization**: Each feature story MUST include:
  - A primary component (page-level or container)
  - 1-N presentational/reusable sub-components
  - Optional custom hooks if state logic exceeds 50 lines
  - One `.module.css` or `.styled.ts` file per component unless using Tailwind

## Testing Requirements for React

Verification-First principle applies rigorously to React components:

- **Unit Tests** (required for all components and hooks):
  - Library: Vitest or Jest
  - Rendering: React Testing Library (NOT enzyme or Shallow rendering)
  - Coverage: Minimum 80% for business logic; 100% for critical user paths
  - Pattern: Test behavior, not implementation details
- **Integration Tests** (required for feature stories):
  - Test user workflows spanning 2+ components
  - Mock external API calls with MSW (Mock Service Worker)
  - Cover form submission, navigation, data loading sequences
- **E2E Tests** (required for P1 user stories):
  - Tool: Playwright or Cypress
  - Scenarios: Complete user journeys from landing to success state
  - Minimum: One E2E test per P1 story
- **Accessibility Tests** (required):
  - Use `@axe-core/react` or `jest-axe` in unit tests
  - Manual WCAG 2.1 Level AA spot-check for each story
  - No new components without accessible ARIA labels

Test files MUST follow naming convention: `ComponentName.test.tsx` and live alongside the component.

## Performance & Accessibility Standards

React applications MUST meet these non-negotiable standards:

- **Performance Gates**:
  - Core Web Vitals: LCP < 2.5s, FID < 100ms, CLS < 0.1 (or INP < 200ms)
  - Bundle size: Warn at 250 KB gzipped, fail at 500 KB for initial bundle
  - Time to Interactive: < 3s on 4G / Moto G4
  - Lighthouse score: Minimum 85 on each story before merge
- **Code Splitting**: Every route/lazy component MUST use React.lazy() with Suspense
- **Image Optimization**: All images MUST use next/image or <img with srcSet> + lazy loading
- **Accessibility** (WCAG 2.1 AA minimum):
  - All interactive elements MUST be keyboard navigable (Tab, Enter, Escape)
  - Form inputs MUST have associated `<label>` elements
  - Images MUST have `alt` text or `aria-hidden="true"` if decorative
  - Color contrast MUST meet AA standards (4.5:1 for text)
  - Headings MUST follow semantic hierarchy (h1, h2, h3...)
- **SEO Basics** (if server-rendered or SSG):
  - Metadata: document title, description, canonical URL per page
  - OpenGraph and Twitter Card tags for social sharing
  - sitemap.xml and robots.txt required for production

Violations documented in story acceptance criteria or corrected before PR merge.
